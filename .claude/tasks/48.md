# Task 48 — Add /unity-mcp-setup — detect, install, and register the Unity MCP plugin, and record it per-project

Target: claude

## Goal
Ship a new `/unity-mcp-setup` command that makes a Unity project ready for
MCP-assisted task implementation. It is the single, re-runnable, idempotent
entry point that (a) confirms the project is Unity, (b) detects/installs the
Unity-side package `com.coplaydev.unity-mcp` in `Packages/manifest.json`,
(c) records the project-side fact in versioned artifacts, and (d) registers
and verifies the Claude-side MCP server on this machine. It configures only
what is missing and can carry a from-nothing project all the way to a
tested connection.

## Acceptance criteria
- New `commands/unity-mcp-setup.md` with full frontmatter
  (name: unity-mcp-setup, version: 0.1.0, type: command, description).
- Refuses on non-Unity projects: probes `ProjectSettings/ProjectVersion.txt`
  read-only (same signal /project-setup uses) and stops cleanly if absent.
- Unity-side: reads `Packages/manifest.json`; if `com.coplaydev.unity-mcp`
  is absent, offers to add it (idempotent — no change if already present) by
  inserting this exact dependency entry into the `dependencies` object:
  `"com.coplaydev.unity-mcp": "https://github.com/CoplayDev/unity-mcp.git?path=/MCPForUnity#main"`.
  On a from-nothing project, instructs the user to open the Unity Editor so
  the MCP server starts.
- Versioned project-side retention:
  - Writes/updates a terse marker line in CLAUDE.md exactly:
    `Unity MCP for /task-implement: com.coplaydev.unity-mcp (UnityMCP, http)`
    (updates in place, never duplicates).
  - If the project has a context layer (`.claude/context/INDEX.md` exists),
    creates `.claude/context/mcp-tools.md` — Unity-only — documenting the
    `claude mcp add --transport http UnityMCP http://127.0.0.1:8080/mcp`
    registration, the `mcp__UnityMCP__*` tool namespace, the
    `mcpforunity://…` resources, the alternate-port note, and the
    "running session's tool index doesn't refresh after add" gotcha; and
    adds a corresponding row to `.claude/context/INDEX.md`. If there is no
    context layer, the CLAUDE.md marker alone is written (no orphan file).
- Claude-side retention leans on the native mechanism (no custom cache
  file): checks `claude mcp list`; if `UnityMCP` is not registered, runs
  `claude mcp add --transport http UnityMCP http://127.0.0.1:8080/mcp`
  (local scope → per-machine, per-project, unversioned in ~/.claude.json).
- Handles the tool-index-refresh gotcha: after a fresh `claude mcp add`,
  tells the user to restart the session, then confirms on re-run; if
  `UnityMCP` is already registered+connected, verifies immediately via
  `claude mcp list` and reports connected/disconnected (disconnected →
  reminds the user to open Unity).
- Idempotent end-to-end: a second run on a fully-configured project makes
  no changes and reports "already set up".
- Feature is documented: `.claude/context/features.md` overview, README
  feature list, and `docs/cli-help.txt` updated; root VERSION bumped (minor).

## Decisions
- Dedicated command rather than folding detection into /project-setup: the
  Claude-side extension is usually wired up after first-time init and the
  step must be independently re-runnable. /project-setup only *offers* to
  run it (task 50).
- Claude-side (per-machine, per-project, unversioned) state is provided
  natively by `claude mcp add` local scope in ~/.claude.json and queried by
  `claude mcp list`; no bespoke cache file is invented.
- Project-side (versioned) facts split into a terse CLAUDE.md marker (cheap,
  auto-loaded every session) plus a richer `.claude/context/mcp-tools.md`
  when a context layer exists (on-demand detail + onboarding doc).
- Detection signal for "is Unity" is ProjectVersion.txt (matches
  /project-setup); manifest.json is used only to detect/insert the package.
