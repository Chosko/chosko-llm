# Task 14 — task-enrich — new skill to expand thin tasks for local LLM

Target: claude

## Goal

Create a new `task-enrich` skill that takes an existing thin (`target: claude`)
task body and transforms it into a self-contained enriched body suitable for a
local LLM implementer. The skill reads the thin body, navigates the project's
source files and context layer to gather everything the local LLM will need,
then rewrites the body in place adding two new sections: `## Context bundle` and
`## Implementation steps`. It sets the `Target:` field to `local` in the
rewritten body.

## Acceptance criteria

- New skill exists at `skills/task-enrich/SKILL.md` with valid frontmatter
  (`name: task-enrich`, `version: 0.1.0`, `type: skill`, `description`).
- Smoke test exists at `tests/smoke/task-enrich.md`.
- The skill is project-agnostic: no hardcoded file paths, project names, or
  tool names. It navigates whatever context layer the project has.
- Enrichment adds exactly two sections to the thin body without modifying Goal,
  Acceptance criteria, Decisions, or Hints:
  - `## Context bundle` — relevant file excerpts, patterns, and constraints
    the local LLM needs; nothing beyond what is necessary.
  - `## Implementation steps` — concrete step-by-step guidance; enough for a
    model that cannot make additional reads.
- `Target:` field in the body file is updated from `claude` to `local`.
- If the task is already `target: local`, the skill warns and exits without
  re-enriching.
- `.claude/context/features.md` entry added for `task-enrich`.
- `.claude/domain/task-workflow.md` cross-references `task-enrich`.

## Decisions

- Enrichment is in-place: the body file is rewritten, not duplicated. There is
  no separate "enriched copy" — the body transitions from thin to enriched.
- The skill does not commit after enriching. Committing is the user's
  responsibility (or handled by a wrapper the user chooses). This keeps the
  skill single-responsibility.
- Context bundle should be selective: include what the local LLM needs, exclude
  what it doesn't. A bloated bundle defeats the purpose (context overflow is the
  very problem we are solving).
- Implementation steps should be concrete enough to follow without any external
  reads. If a step relies on a pattern not in the context bundle, that pattern
  must be added to the context bundle.
