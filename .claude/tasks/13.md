# Task 13 — task schema — introduce thin body format and target field

Target: claude

## Goal

Replace the old long-form body schema (Description, Required reading, Relevant
snippets, Conventions to follow, Out of scope, Definition of done, etc.) with a
lean schema optimised for Claude as implementer. The new schema has four
sections: Goal, Acceptance criteria, Decisions, and Hints. Add a `target` field
(values: `claude` | `local`, default `claude`) to both the body file and the
TASKS.md summary block. Update the domain doc so every other task-* command and
skill can reference it as the single source of truth.

## Acceptance criteria

- `.claude/domain/task-workflow.md` documents the new thin body schema with all
  four sections, their purpose, and whether each is required.
- `target` field is defined: `claude` (default, thin body) and `local`
  (enriched body written for a local LLM).
- `Hints` is documented as required — always present, even if the list is short.
- `Decisions` is documented as conditional — only included when non-obvious
  choices were made during authoring (by the user or by Claude).
- The old schema sections are removed from the doc; no "legacy" alias is kept.
- The TASKS.md summary block format is updated to include `Target:` alongside
  `Status:`, `Files:`, and `Preconditions:`.
- The doc notes that `target` lives in both the body file header and the
  TASKS.md summary block (the only intentionally duplicated field besides
  `Files:`).

## Decisions

- `Hints` is required, not optional. If nothing collateral exists, write
  `Hints: none` explicitly rather than omitting the section, so implementers
  know the author considered it.
- `Decisions` is omitted entirely when there are no non-obvious choices — its
  absence is meaningful (no contested calls were made).
- `target` is mirrored in both the body file and TASKS.md so the backlog
  view shows implementer intent without opening body files.
- The old schema is removed without a deprecation period: all existing body
  files pre-date this task and were authored under the old contract; they stay
  as-is and are not retroactively reformatted.
- No YAML frontmatter in body files — `Target:` is a plain inline field on the
  second line of the body file, consistent with how TASKS.md uses `Status:` and
  `Files:` as plain key-value fields rather than YAML.

## Hints

- .claude/domain/task-workflow.md — primary file to rewrite
- .claude/TASKS.md — add `Target:` field to the summary block format description
  (the four tasks 13–16 already use it as a forward example)
