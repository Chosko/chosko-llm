# Task 38 — Split the dirty-tree "proceed" option in /task-implement into commit vs. leave-uncommitted

Target: claude

## Goal
When `/task-implement` finds a dirty working tree before starting a run, it
currently offers a 3-way prompt: proceed (which silently folds the unrelated
dirty changes into the task's own commit), commit first, or abort. Folding
into the task's commit is not what most "proceed" callers actually want —
it conflates unrelated changes with the task's own change set. Split
"proceed" into two explicit choices so the user decides up front whether
the pre-existing dirty changes ride along in the task's commit or stay
exactly as they are in the working tree.

## Acceptance criteria
- The dirty-tree prompt in `commands/task-implement.md` (PRE-FLIGHT CHECKS,
  section 1) presents four options instead of three:
  1. proceed, leave uncommitted — run the task; the pre-existing dirty
     changes are left untouched in the working tree and are NOT part of
     Step 8's commit.
  2. proceed, commit together — run the task; the pre-existing dirty
     changes are staged and folded into the task's own Step 8 commit.
  3. commit first — commit the pre-existing dirty changes now, before the
     task runs (same behavior as today's option 2, renumbered).
  4. abort — stop now, leave everything as-is (same as today's option 3,
     renumbered).
- Typed single-letter/word answers are accepted for all four (case
  insensitive); EOF, empty line, or an unrelated reply is still treated as
  abort, unchanged from today.
- Choosing option 2 asks about untracked files the same way option 3
  (commit first) already does — "Also fold untracked files into the task's
  commit? [y/N]" — and the answer is recorded for Step 8.
- Step 8 (Commit) is updated: when the run's pre-flight choice was option 2,
  Step 8 stages the pre-existing tracked dirty files (via `git add -u`,
  which also covers the task's own tracked edits) plus, if the user opted
  in, the pre-existing untracked files explicitly by path — in addition to
  the task's own files. When the pre-flight choice was option 1 (or the
  tree was clean), Step 8's staging is unchanged: only the task's own files
  by explicit path, exactly as today.
- This choice only ever applies to the first task in a run (the dirty-tree
  check runs once, before any task) — later tasks in the same invocation
  are unaffected.
- When `--no-commit` is passed, option 2 is not offered (there is no task
  commit for it to fold into) — the prompt reverts to a 3-way choice:
  proceed/leave-uncommitted, commit first, abort.
- `tests/smoke/task-implement.md`'s "Dirty-tree" scenarios are updated to
  cover: proceed-leave-uncommitted (renamed from today's "proceed"),
  proceed-commit-together (new), commit-first (renumbered), abort
  (renumbered), and the `--no-commit` 3-way variant.

## Decisions
- Under `--no-commit`, option 2 ("commit together") is dropped rather than
  kept-but-inert, because offering a choice that silently does nothing
  under that flag would be confusing. The prompt degrades to today's
  original 3-way shape in that mode.
- Step 8's folding uses `git add -u` (not explicit per-path staging) when
  option 2 is chosen, because it needs to catch both the task's own tracked
  edits and the pre-existing tracked dirty files in one commit without
  duplicating path lists; untracked files are still staged explicitly by
  path, matching the existing "commit first" flow's pattern.
