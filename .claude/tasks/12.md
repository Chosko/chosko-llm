# Task 12 — task-clean — auto-commit pruned tasks after apply

## Description

`/task-clean` currently ends after PHASE 2 (apply) with no git interaction — the
command explicitly states "Commit. This command does not touch git." and the smoke
test asserts "No git commit is made." This is inconsistent with the `task-add`
v0.5.0 behaviour, which auto-commits the two written files without prompting.

This task adds a **PHASE 3 — COMMIT** to `/task-clean` that runs automatically after
PHASE 2 succeeds. No additional user prompt is needed. The phase stages exactly the
files it touched — `.claude/TASKS.md` (modified) and each deleted body file — then
commits them with a conventional message.

Staging a deleted file works identically to staging a modified one: `git add --
path/to/deleted.md` stages the deletion when the file no longer exists on disk.

The command's version bumps from `0.2.0` → `0.3.0` (minor — new observable behaviour).
The frontmatter `description` field is updated to mention the auto-commit. The
"DO NOT … Commit. This command does not touch git." prohibition in the DO NOT list
is removed and the tool-discipline note is updated to acknowledge that Bash is now
also used for the git operations.

The smoke test `tests/smoke/task-clean.md` is updated: the "No git commit is made"
assertion becomes "A git commit is made automatically after pruning" and two new
checklist items verify the commit.

### Conventions to follow

- No new external dependencies; the Bash / PowerShell tool is used only for git
  operations and the existing `rm` calls.
- Commit message must follow the project's conventional style:
  `task-clean: remove tasks <N>[, <M>, …]` — matching the existing commit-history
  pattern visible in git log (e.g. `task-clean: remove body files for tasks 8 and 9`).
- Use `git add --` with explicit file paths only. Never use `git add -A`,
  `git add .`, or `git add -u`. Stage both `.claude/TASKS.md` and each deleted body
  file path (deleted files are staged the same way as modified ones via `git add --`).
- Do NOT use `--amend`, `--no-verify`, `--no-gpg-sign`, or any hook-skipping flag.
- On commit failure: surface the raw output, leave files in their current git state,
  and let the user decide. Do not retry.
- PHASE 3 stages ONLY the files PHASE 2 modified or deleted.

### Behavior change

**Before:** `/task-clean` ends after PHASE 2 with no git interaction. The user must
commit manually.

**After:** `/task-clean` automatically runs PHASE 3 — COMMIT after PHASE 2 succeeds.
It stages `.claude/TASKS.md` plus all deleted `.claude/tasks/<N>.md` paths and
commits with a message of the form `task-clean: remove tasks <N>[, <M>, …]`. The
commit hash is reported. On failure the raw error is surfaced and no retry is
attempted.
