# Task 10 — cmd-ls — add STATUS column with up-to-date / updatable / not-installed / local-only labels

## Description

`chosko-llm ls` currently prints four columns: `NAME KIND INSTALLED LATEST`.
It computes both version strings but shows no derived status, so the user has
to mentally compare two values on every row to decide whether a feature needs
action. This task adds a fifth `STATUS` column to the right of `LATEST` that
summarises the comparison in a human-readable label:

| INSTALLED | LATEST | STATUS         |
|-----------|--------|----------------|
| —         | 0.3.0  | not installed  |
| 0.3.0     | —      | local only     |
| 0.3.0     | 0.3.0  | up-to-date     |
| 0.2.1     | 0.3.0  | updatable      |

The STATUS value is derived entirely from the values of `inst_col` and
`latest_col` already computed inside `list_all` for each row. No new file I/O
is needed.

The `—` sentinel is the string the script uses for a missing version today;
the comparison must treat it as "no version", not as a version identifier.

### Behavior change

After this task lands, `chosko-llm ls` output gains a fifth column `STATUS`:

```
NAME                           KIND     INSTALLED      LATEST           STATUS
task-add                       skill    0.5.0          0.5.0            up-to-date
task-implement                 skill    0.4.0          0.5.0            updatable
old-tool                       command  0.1.0          —                local only
refactor-codebase              skill    —              0.3.0            not installed
```

The `--installed` and `--available` filters continue to operate on the
INSTALLED/LATEST values as before; the STATUS column is informational only.

The `print_header` function and the two `printf` format strings (commands loop
and skills loop) must be widened to accommodate the new column.

Suggested column widths: `%-30s %-8s %-14s %-16s %s` (STATUS last, no
padding needed).
