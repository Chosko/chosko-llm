# Task 25 — Add a `--commit` flag to commands that leave changes uncommitted

Target: claude

## Goal
Give each authoring command that currently leaves output uncommitted an opt-in
`--commit` flag that commits the changes it produced at the end. Default (no
flag) behavior is unchanged. Commands in scope: `/context-build`,
`/task-enrich`, `/refactor-codebase`, `/refactor-tests`, `/task-setup`, and
`/project-setup`.

## Acceptance criteria
- Each in-scope command parses `--commit` from its arguments and, when present,
  commits the changes it produced at the end, staging only the explicit paths
  it wrote (no catch-all).
- Without `--commit`, each command behaves exactly as before (uncommitted).
- `/project-setup --commit`: commits its OWN artifacts first, then invokes the
  nested commands (`/task-setup`, `/context-build`) WITH `--commit` so each
  commits its own output.
- `--commit` and `--no-commit` (task 26) are mutually exclusive — passing both
  is an error with a clear message.
- Commit messages follow each command's natural subject and the repo's existing
  commit style; no `--no-verify` / `--amend` / hook-skipping; no push/branch/tag.
- For a non-git VCS, `--commit` honors the `CLAUDE.md` `## VCS` mapping
  (git→cm).
- Each touched command's smoke test and its entry in
  `.claude/context/features.md` are updated; each touched command's `version`
  is bumped per the authoring guide.

## Decisions
- Shared flag convention with task 26 (same parsing, wording, and mutual
  exclusivity). See [[task-26]] for the auto-commit side.
- task-setup is in scope only because task 23 made it uncommitted-by-default.
- project-setup is included per the user's decision: with `--commit` it commits
  its own artifacts, then delegates `--commit` to the nested commands.
