# Task 29 — Add a "Workflows" section to README before "Development"

Target: claude

## Goal
Give a newcomer a narrative map of the *ways of working* the shipped
commands enable, rather than a per-command reference. Insert a new
top-level section immediately before `## Development` in `README.md` that
explains the four workflows by intent and when you'd reach for them:
project setup, navigation context, refactoring, and the task backlog. The
README already documents the `chosko-llm` CLI verbs; this section is about
the installable Claude Code commands and how they fit together.

## Acceptance criteria
- A new top-level section (e.g. `## Workflows`) sits immediately before the
  existing `## Development` heading.
- It reads as a connected narrative — a short intro plus one short
  paragraph per workflow — explaining intent and usage, not enumerating
  every flag:
  - **Project setup** — `/project-setup` as a one-shot first-time bootstrap
    (CLAUDE.md seeding, AGENTS.md, VCS mapping, optionally the task backlog
    and context layer).
  - **Navigation context** — `/context-build` to create a token-cheap
    codebase summary layer, `/context-update` to keep it fresh after
    changes.
  - **Refactoring** — `/refactor-codebase` and `/refactor-tests` as
    behaviour-preserving, plan-first, test-gated cleanups.
  - **Task backlog** — the `/task-setup` → `/task-add` → `/task-implement`
    → `/task-list` / `/task-clean` loop, with `/task-enrich` for handing a
    task to a local LLM.
- Mentions, once, the shared `--commit` / `--no-commit` convention so the
  reader knows commit behaviour is controllable.
- Notes these commands are opt-in installable features (`chosko-llm add`),
  consistent with the rest of the README.
- Explanatory but concise — no behaviour changes, docs only.

## Decisions
- Frame each workflow around intent and when-you'd-use-it, as if teaching a
  new method of working — not as a command-by-command flag reference.
- Keep it tight: brief intro + one compact paragraph per workflow.

## Hints
- README.md — insert the new section directly above `## Development`
  (around the existing `---` / Development boundary). Match the README's
  existing heading style and tone.
