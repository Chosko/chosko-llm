# Task 5 — Task workflow — `/task-implement` prompts on dirty tree instead of hard-aborting

## Description
Today, `commands/task-implement.md` PRE-FLIGHT CHECKS step 1 hard-
refuses to run if `git status --porcelain` reports any uncommitted
changes:

> Working tree must be clean. Run `git status --porcelain`. If
> anything is uncommitted (other than what `.gitignore` excludes),
> stop and report. The user must commit or stash first — this
> command will not mix in unrelated changes.

The same rule appears at BETWEEN TASKS step 1 ("Confirm the working
tree is clean again (`git status`).").

This is friction in the common case where the user has small in-
progress edits they don't want to throw away or stash. Replace the
hard abort with an interactive 3-way prompt:

  1. **Proceed anyway** — continue with the task even though
     uncommitted changes exist. The risk (commit will mix unrelated
     changes) becomes the user's choice.
  2. **Commit first** — `/task-implement` asks the user for a commit
     message, stages the currently dirty files (`git add -u` plus
     any new untracked files the user explicitly opts in to), makes
     a commit, then proceeds with the task.
  3. **Abort** — what the current behavior does. Stop, no changes.

### Files to modify
commands/task-implement.md, tests/smoke/task-implement.md

### Required reading

- commands/task-add.md (after Task 4 lands) — gains an analogous "offer to commit" phase. The dirty-tree prompt here should feel like the same family of UX, but they are independent.
- tests/smoke/task-implement.md — extend with new dirty-tree cases.
- docs/authoring-guide.md — version-bump rules: behavior change to an existing command → minor bump.
- CLAUDE.md — frontmatter is mandatory; `version` must be bumped.

### Conventions to follow
- Tool discipline already documented in `task-implement.md`: file reads via Read; file edits via Edit; new files via Write. Bash is permitted only for git, tests, and the existing flow.
- Never use `--no-verify`, `--amend`, `--no-gpg-sign`, or skip hooks for the pre-task commit, mirroring Step 8.
- Never run destructive git operations (`reset --hard`, `clean -f`, `checkout .`, `stash drop`) without explicit user instruction. The new prompt must NOT offer a "stash" path silently — if the user wants stash, they can pick "abort" and stash by hand.
- The prompt must be explicit and require a typed answer (e.g. `1`/`2`/`3` or `proceed`/`commit`/`abort`). Silence ⇒ abort. Match the posture of Step 8: never act on shared/visible git state without explicit approval.
- Frontmatter `version` must bump (per CLAUDE.md hard rule + authoring-guide). Current is 0.3.0; bump to 0.4.0 (additive UX change to existing command).

### Out of scope
- Auto-stash. The 3 options are exactly: proceed, commit, abort.
- Changing Step 8 (the per-task commit). That stays one-commit-per-task.
- Changing `/task-add`, `/task-clean`, `/task-setup`, `/task-list`.
- Pushing, branching, tagging.
- Changing FAILURE HANDLING semantics — a halted run still leaves status `[IN PROGRESS]`.
- Behavior on a `.gitignore`-excluded dirty file: those are still ignored as today.

### Behavior change
- **PRE-FLIGHT CHECKS step 1** is rewritten:
  - Run `git status --porcelain` once.
  - If output is empty (clean tree), continue silently.
  - If non-empty, list the dirty files to the user (truncated to first ~20 entries with a "(+N more)" tail) and prompt:
    ```
    Working tree has uncommitted changes. Choose:
      [1] proceed   — run the task anyway (changes will be mixed into the task's commit)
      [2] commit    — commit the current changes first, then run the task
      [3] abort     — stop now, leave everything as-is
    ```
  - On `1` (proceed): warn that "Step 8 (Commit) will include these unrelated changes in this task's commit", then continue.
  - On `2` (commit): ask "Commit message?" and read a single-line answer (or a multi-line answer terminated by an empty line). Stage tracked dirty files (`git add -u`); for any untracked files in the porcelain output, list them and ask "Also include untracked? [y/N]" — only stage them if the user says yes. Create one commit using the user's message via HEREDOC. Then continue with the task.
  - On `3` (abort) or any other answer / EOF / silence: stop. No state changes.
- **BETWEEN TASKS step 1** uses the same prompt. A dirty tree between tasks is unusual (the previous task should have committed everything it changed) but the same 3 options apply consistently.
- The current sentence "this command will not mix in unrelated changes" is removed from PRE-FLIGHT CHECKS — option 1 explicitly allows it, with the user's eyes open.
- Frontmatter `version: 0.3.0` → `0.4.0`; `description` updated to mention the dirty-tree prompt.

### Tests
- `tests/smoke/task-implement.md` — extend the manual smoke checklist:
  - **Clean tree (regression)**: clean working tree, run `/task-implement <N>`, no prompt fires, normal flow runs.
  - **Dirty tree → proceed**: leave a tracked file dirty, run `/task-implement`, choose `1`, confirm warning is printed, the task proceeds, and Step 8's commit DOES include the unrelated change.
  - **Dirty tree → commit (tracked only)**: leave a tracked file dirty, run, choose `2`, supply a commit message, decline untracked inclusion, observe a single pre-task commit with that message and only the tracked dirty file, then the task runs and produces its own separate commit.
  - **Dirty tree → commit (with untracked)**: leave both a tracked dirty file and an untracked file, run, choose `2`, accept untracked inclusion, observe both files in the pre-task commit.
  - **Dirty tree → abort**: leave a dirty file, run, choose `3`, observe no git operations and no `Status:` flips.
  - **Pre-commit hook failure on option 2**: simulate a failing hook, choose `2`, observe the failure surfaces, no `--no-verify` retry, and the run halts before any task work begins.
  - **Between-tasks dirty**: with `/task-implement` invoked on multiple tasks, contrive a scenario where the working tree is dirty between tasks (e.g. a generated file written by Step 4 is not committed) and confirm the same 3-way prompt fires before the next task starts.
