# Task 34 — Add a `--version` flag to the base `chosko-llm` command

Target: claude

## Goal
There is no way to ask the installed CLI what version it is. `install.sh`
computes and prints a version at install time, but the running proxy never
exposes it. Add `chosko-llm --version` (and the usual aliases) so a user can
check the installed version at any time. The output mirrors the install-time
format: the `VERSION` file contents plus the managed clone's `git describe`,
e.g. `0.1.0 (v0.1.0-3-gabc123)`.

## Acceptance criteria
- `chosko-llm --version` prints the version and exits 0, handled by the proxy
  alongside the existing `help` dispatch (before the subcommand `case`'s
  unknown-subcommand path).
- `-v` and the bare word `version` are accepted as aliases for the same
  behavior.
- The version string matches install.sh's format: the trimmed contents of
  `$CHOSKO_LLM_HOME/VERSION`, plus ` (<git describe>)` when git and a describe
  value are available; falls back to `unknown` if VERSION is missing.
- The version-resolution logic is not duplicated by copy-paste between
  install.sh and the proxy — factor a shared helper (see Decisions) so the two
  stay in sync.
- `chosko-llm help` / `docs/cli-help.txt` document the `--version` flag.
- README documents `chosko-llm --version`.

## Decisions
- Format mirrors install.sh (`VERSION` + git describe), confirmed with the
  user — consistency with what install already prints.
- Dedupe approach: add a `resolve_version` helper to `scripts/lib.sh` and have
  both the proxy path and install.sh use it. The proxy is intentionally minimal
  and does not currently source lib.sh; the implementer decides the
  least-invasive wiring (e.g. a tiny `scripts/cmd-version.sh` that sources
  lib.sh and is exec'd by the proxy, OR sourcing only the helper). Whatever is
  chosen, the resolution logic must live in exactly one place.
- `version` as a bare subcommand is added in addition to the `--version`/`-v`
  flags so it reads naturally next to `help`.

## Hints
