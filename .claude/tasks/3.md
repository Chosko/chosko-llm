# Task 3 — Task workflow — automate the 8-step `/task-implement` sequence for external LLMs (aider + Ollama)

## Description
Today, `/task-implement` (run by Claude Code) executes a strict 8-step
TDD-style sequence per task: mark IN PROGRESS → write tests → run and
watch fail → implement → run and watch pass → run full suite → mark
DONE → commit. Task 1 produced `.claude/external/implement-prompt.md`
so an external LLM (qwen2.5-coder:14b via aider) can implement a single
task body, but the surrounding orchestration is missing — the user
still has to flip statuses in `TASKS.md`, run tests, retry on failure,
and commit by hand. This task adds that orchestration so external-LLM
implementation gives the same end-to-end experience as the Claude path.

The solution has two layers, per the design discussion:

1. **Shared orchestration in the chosko-llm CLI.**
   - `scripts/lib-task-external.sh` (new) — shared bash helpers:
     parse a task's summary block out of `.claude/TASKS.md`
     (status, files, preconditions, title); flip a task's `Status:`
     line atomically; locate the task body file at
     `.claude/tasks/<N>.md`; emit log lines via the existing
     `lib.sh` logger.
   - `scripts/cmd-task-impl.sh` (new) — orchestrator. Driven by
     `chosko-llm task-impl <N>` (also accepts a list and `all`,
     same arg semantics as `/task-implement`). Performs the 8-step
     sequence end-to-end against a single project. Sources both
     `lib.sh` and `lib-task-external.sh`.
   - `bin/chosko-llm` — dispatch the new `task-impl` subcommand.
   - `docs/cli-help.txt` — document `task-impl` alongside the other
     subcommands.

2. **Per-project wiring written by `/task-setup`.**
   - `/task-setup` gains a new responsibility: at run time, infer how
     this project's tests should be invoked, then write two thin
     wrapper scripts under `.claude/external/`:
       - `run-affected-tests.sh <test-file> [<test-file>…]` — runs
         the test runner against the listed files only.
       - `run-full-tests.sh` — runs the project's full test suite.
     The orchestrator (`cmd-task-impl.sh`) calls these wrappers; it
     does no test-runner detection of its own. This keeps the
     project-specific knowledge in the project.
   - Inference uses the same heuristics as `/task-implement`'s
     LOCATING THE TEST RUNNER section (pytest, npm test, cargo
     test, go test, Makefile target). If the project has no
     detectable test runner, `/task-setup` prompts the user in
     `no-test-suite mode` with two options:
       - **(A) Halt** — refuse to write the wrapper scripts; tell
         the user to set up a test suite and re-run `/task-setup`.
         `chosko-llm task-impl` will refuse to run without the
         wrappers in place.
       - **(B) Skip tests** — write the wrapper scripts as no-op
         stubs (exit 0 with a clear message). The orchestrator
         detects the no-op stub and switches to skip-tests mode:
         skips steps 2/3/5/6, requires per-task confirmation, and
         appends `(no tests — manual verification pending)` to
         commit messages.
   - `/task-setup` remains idempotent. Re-running on a project that
     was previously set up without a test suite (mode B) but has
     since added tests must re-detect the runner and overwrite the
     no-op stubs with real wrappers, after confirming with the user.

3. **Two-pass aider sequence** in the orchestrator. For each task:
   - Step 1: flip `Status:` to `[IN PROGRESS]` in `.claude/TASKS.md`.
   - Step 2 (tests pass): invoke aider with a new prompt at
     `.claude/external/tests-prompt.md` (written by `/task-setup`,
     analogous to `implement-prompt.md`) instructing the model to
     write/extend ONLY the test files for the task. After the run,
     call `run-affected-tests.sh` against the task's test files;
     expect a non-zero exit (tests must fail).
   - Step 4 (impl pass): invoke aider with the existing
     `implement-prompt.md`. Run `run-affected-tests.sh`; on
     failure, re-invoke aider up to N times (default N=3,
     overridable via env var `CHOSKO_TASK_IMPL_RETRIES`), each
     retry feeding the failure log back to aider via `--message`.
   - Step 6: run `run-full-tests.sh`. On failure, retry the impl
     pass under the same N-retry budget. On exhausted retries,
     leave status `[IN PROGRESS]` and stop the run (no commit).
   - Step 7: flip `Status:` to `[DONE]`.
   - Step 8: stage modified files (Files-list ∪ TASKS.md) and
     create one commit, message `Task <N>: <title>` plus body. In
     skip-tests mode, append the parenthetical note.

### Conventions to follow
- POSIX-ish bash only: no jq/yq/python (CLAUDE.md hard rule).
- Every new script under scripts/ starts with `set -euo pipefail` and sources `lib.sh`.
- All scripts honor `CHOSKO_LLM_HOME` and `CLAUDE_HOME`; never hardcode `~/.chosko-llm` or `~/.claude`. Use helpers in `lib.sh`.
- Frontmatter: `commands/task-setup.md` keeps `name`, `version`, `type`, `description`. Bump `version` per docs/authoring-guide.md (this is an additive feature on an existing command — minor bump from 0.2.0 → 0.3.0).
- File reads via Read tool; file edits via Edit; new files via Write. No shell redirection for content.
- Filesystem is the source of truth. No lockfile, no state file.
- One commit per task (`bin/chosko-llm` + `docs/cli-help.txt` changes ride along in this task's commit).

### Out of scope
- Changes to `commands/task-implement.md` — the Claude-driven path is unchanged.
- Changes to `commands/task-add.md` or `commands/task-clean.md`.
- Adding new external dependencies (no jq, no python). No new deps for aider beyond what the user already has.
- Symlink-based wrappers. `.claude/external/run-*.sh` are real files written by `/task-setup`.
- Lockfiles or persisted state for retry budgets — pass via env var only.
- Auto-installing aider or Ollama. Assume the user has them; orchestrator should fail loudly with a clear message if `aider` is not on PATH.
- Cross-project orchestration (running tasks from multiple projects in one invocation).

### Behavior change
- New global subcommand: `chosko-llm task-impl <N> [<N>…] | all`. Runs the 8-step sequence per task against the current project; commits each task separately.
- `/task-setup` now also: (a) infers the project's test runner; (b) writes `.claude/external/run-affected-tests.sh` and `.claude/external/run-full-tests.sh`; (c) writes `.claude/external/tests-prompt.md` (the test-pass analogue of `implement-prompt.md`). All three are idempotent — existing files are not overwritten unless the user confirms (e.g. moving from no-test-suite stubs to real wrappers).
- `/task-setup`'s no-test-suite handling matches `/task-implement`'s: prompt user with (A) halt or (B) write no-op stubs and run in skip-tests mode.
- Orchestrator refuses to run if any of the three artifacts is missing — points the user to `/task-setup`.
- Working tree must be clean before each task; orchestrator stops on dirty tree.
- On test failure during the impl pass, orchestrator re-invokes aider up to `CHOSKO_TASK_IMPL_RETRIES` (default 3) times with the failure log appended. On exhaustion, status stays `[IN PROGRESS]` and the run halts.
- Skip-tests mode commits get `(no tests — manual verification pending)` appended to the message body.

### Tests
- `tests/smoke/cmd-task-impl.md` (new): manual smoke checklist covering
  - happy path: 1-task implementation with passing tests, single commit appears, status flipped to `[DONE]`;
  - retry path: simulate a flaky impl by tampering, observe up to N retries before halt;
  - skip-tests path: project with stub wrappers, observe per-task confirmation prompt, commit message includes the parenthetical note;
  - dirty-tree refusal;
  - missing-artifacts refusal (e.g. delete `run-full-tests.sh`, run, expect a clear error pointing to `/task-setup`).
- `tests/smoke/task-setup.md` (extend): add cases for
  - first-time inference on a pytest project (creates real wrappers);
  - no-test-suite project, choose A (halt, no wrappers written);
  - no-test-suite project, choose B (no-op stubs written);
  - re-running `/task-setup` after the user adds a test suite to a previously stub-mode project (real wrappers replace stubs after confirmation);
  - idempotency: re-running on a fully set-up project leaves all artifacts untouched.
