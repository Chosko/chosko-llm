# Task 4 — Task workflow — `/task-add` offers to commit the new task at the end

## Description
Today, `commands/task-add.md` ends PHASE 4 with an explicit
`DO NOT: Commit. This command does not touch git.` rule, so after
the task index and body are written the user must remember to stage
and commit them by hand. This is friction: the two files written by
`/task-add` (`.claude/TASKS.md` + `.claude/tasks/<N>.md`) are a
natural, self-contained commit, and the user almost always wants to
capture them as one before moving on.

Change `/task-add` to add a final phase that asks whether to commit.
The command must remain safe with a dirty working tree — it must
stage ONLY the two files it just wrote, never `-A`/`-u`, and never
amend or push.

### Files to modify
commands/task-add.md, tests/smoke/task-add.md

### Required reading
- commands/task-add.md (full file) — current PHASE 4 and the trailing `DO NOT: Commit.` rule both need to change. Frontmatter version must bump 0.3.0 → 0.4.0 (additive feature).
- commands/task-implement.md (Step 8 — Commit section) — already implements a per-task git commit pattern (HEREDOC message, no `--no-verify`/`--amend`/hook skipping). Mirror that style for consistency.
- tests/smoke/task-add.md — extend with new commit-prompt cases.
- docs/authoring-guide.md — version-bump rules (additive feature → minor bump).
- CLAUDE.md — hard rule: every `commands/<name>.md` keeps full frontmatter (name/version/type/description); the version field is mandatory.

### Conventions to follow
- Tool discipline already documented in `task-add.md`: file reads via Read; file edits via Edit; new files via Write. Bash is permitted ONLY for the new git phase (status check, stage, commit).
- Stage only the two files `/task-add` wrote in PHASE 4. Never `git add -A`, `git add .`, or `git add -u`.
- Never use `--no-verify`, `--amend`, `--no-gpg-sign`, or skip hooks. If a pre-commit hook fails, surface the failure and let the user fix it; do not auto-retry.
- The commit prompt must be explicit ("Commit these two files now? [y/N]") and silence ⇒ no commit. Match `/task-implement`'s posture: never commit without explicit approval.
- Frontmatter `version` must bump (per CLAUDE.md hard rule + authoring-guide).

### Out of scope
- Pushing, pulling, branching, tagging.
- Touching files other than `.claude/TASKS.md` and `.claude/tasks/<N>.md` in the commit.
- Changing the existing PHASE 0–4 flow except to (a) drop the "this command does not touch git" rule and (b) append the new phase.
- Changing `/task-clean`, `/task-implement`, `/task-setup`, `/task-list`.
- Auto-staging unrelated dirty files in the working tree.

### Behavior change
- PHASE 4 (WRITE) is unchanged in what it writes.
- A new **PHASE 5 — OFFER COMMIT** runs immediately after PHASE 4:
  1. Print a one-line summary: "Task <N> written. Commit `.claude/TASKS.md` + `.claude/tasks/<N>.md` now? [y/N]".
  2. If the user answers yes (`y`/`yes`/`commit`/`go`):
     - Stage exactly those two paths (`git add -- .claude/TASKS.md .claude/tasks/<N>.md`).
     - Create one commit with message:
       ```
       Add task <N>: <title>

       <one-line summary derived from the task body's Description first sentence, optional>
       ```
     - Report the resulting commit hash to the user.
  3. If the user answers no (or anything other than an explicit yes), do nothing — the files remain unstaged in the working tree, and the user gets a one-liner: "Skipped commit. Files left unstaged."
- The trailing `DO NOT: Commit. This command does not touch git.` line is removed.
- Add a new line under `DO NOT:` forbidding `git add -A`/`-u`, amend, push, and hook-skipping flags.

### Tests
- `tests/smoke/task-add.md` — extend the manual smoke checklist:
  - **Commit-yes path**: run `/task-add` end-to-end on a clean tree, answer `y` at PHASE 5, observe a single commit containing exactly the two written files; verify `git show --stat HEAD` lists only `.claude/TASKS.md` and `.claude/tasks/<N>.md`.
  - **Commit-no path**: run `/task-add`, answer `n`, observe both files are written but unstaged (`git status` shows them as untracked / modified).

  - **Pre-commit hook failure**: simulate a failing pre-commit hook, answer `y`, verify the command surfaces the failure and does not retry / amend / use `--no-verify`.
