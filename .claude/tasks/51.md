# Task 51 — Add `chosko-llm channel` — point the managed clone at a branch to test unmerged work

Target: claude

## Goal
Let a user switch the managed clone (`$CHOSKO_LLM_HOME`) onto any branch so they
can try features before they land on `master`, then switch back. Add a new
`channel` subcommand backed by `scripts/cmd-channel.sh` with three forms:
`chosko-llm channel` (no arg) prints the branch the clone is currently on;
`chosko-llm channel --list` shows the branches available on `origin` (the
channels you can switch to), marking the current one; `chosko-llm channel
<branch>` fetches origin, checks out `<branch>`, fast-forwards it, and refreshes
the CLI proxy; `chosko-llm channel master` returns to stable. Because
auto-upgrade's `git pull --ff-only` already follows whatever branch is checked
out, no state file is needed — the clone's checked-out branch is the whole
persistence mechanism.

## Acceptance criteria
- `chosko-llm channel` (no args) prints the managed clone's current branch and exits 0.
- `chosko-llm channel --list` (also `-l`) runs `git fetch` then lists the
  branches available on `origin`, marking the currently checked-out one
  (e.g. a `*` or `(current)` tag). Exits 0.
- `chosko-llm channel <branch>` runs, in the managed clone: `git fetch`,
  `git checkout <branch>`, `git pull --ff-only`, then refreshes the proxy at
  `$BIN_DIR/chosko-llm` from the freshly checked-out `bin/chosko-llm` (same
  refresh logic as cmd-upgrade.sh) and `chmod +x`es the clone's scripts.
- After switching, it prints a reminder to run `chosko-llm update --all` to
  deploy the channel's features — but does NOT run `update --all` itself.
- Unknown or non-existent branch fails via `die` with a clear message; the
  clone is left on its original branch (checkout of a missing branch aborts
  cleanly, nothing half-applied).
- `die`s (not a stack trace) when `$CHOSKO_LLM_HOME` is not a git repo, mirroring cmd-upgrade.sh.
- `bin/chosko-llm` routes `channel` → `scripts/cmd-channel.sh` and forwards `$@`.
- The daily auto-upgrade hook skips the `channel` subcommand (added to the
  same skip list as `upgrade`/`help`/`version`/`uninstall` in scripts/auto-upgrade.sh).
- `docs/cli-help.txt` documents `channel` (no-arg, `--list`, and `<branch>` forms) under the upgrade block.
- README and the context layer (new cmd-channel.md, plus cli-entry.md and
  INDEX.md updates) describe the subcommand.
- Root VERSION bumped (minor).

## Decisions
- Dedicated `channel` subcommand, not an `upgrade --branch` flag — gives
  natural no-arg "show current channel" and `--list` "show available channels"
  forms, and keeps upgrade's job single.
- `--list` fetches first so the branch list reflects origin, not a stale local
  view — the point is to discover channels that were just pushed.
- Switching does a full fetch + checkout + pull --ff-only + proxy refresh in one
  shot (a channel switch should leave you fully on the channel), but only
  *suggests* `update --all` rather than running it — feature deployment stays an
  explicit user step, consistent with `upgrade`.
- No new state file. The checked-out branch persists in the clone and
  auto-upgrade's `--ff-only` already tracks it. Returning to stable is just
  `chosko-llm channel master`.
- After a feature branch is merged and deleted upstream, `pull --ff-only` on it
  will fail; the documented recovery is `chosko-llm channel master`.
