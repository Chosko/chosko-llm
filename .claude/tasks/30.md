# Task 30 — Daily auto-upgrade for the `chosko-llm` CLI

Target: claude

## Goal
Make the first `chosko-llm` invocation each calendar day transparently run
`chosko-llm upgrade` before the requested command, so users stay current
without remembering to upgrade. The behavior is governed by an
opt-in/opt-out preference persisted in a gitignored state file inside the
managed clone. Users are opted in at install time and can toggle the
preference at any time. The state file also records when the upgrade last
ran so it fires at most once per day.

## Acceptance criteria
- A gitignored state file under `$CHOSKO_LLM_HOME` (e.g.
  `$CHOSKO_LLM_HOME/.auto-upgrade-state`) stores `enabled=<true|false>` and
  `last_run=<timestamp>`. `.gitignore` ignores it so `git pull --ff-only`
  in the managed clone is never blocked by it.
- The proxy `bin/chosko-llm`, before dispatching the requested subcommand,
  invokes a guarded helper (`scripts/auto-upgrade.sh "$subcommand" || true`)
  that runs `chosko-llm upgrade` at most once per calendar day when the
  preference is enabled. It skips the `upgrade`, `help`, and empty
  subcommands and never recurses into itself.
- `last_run` is stamped to today's date BEFORE the upgrade runs, so a
  failed/again-offline upgrade does not retry on every subsequent command
  that day.
- A missing state file is treated as enabled (opt-in default). `install.sh`
  writes the state file with `enabled=true` and `last_run=<install date>`
  so it does not immediately re-upgrade right after installing.
- When auto-upgrade fires, an interactive suggestion tells the user it can
  be disabled (`chosko-llm upgrade --disable-auto`). The suggestion text is
  gated to an interactive terminal, mirroring the existing `cmd-ls.sh`
  suggestion gating.
- `chosko-llm upgrade` accepts `--enable-auto` and `--disable-auto`
  (mutually exclusive). These ONLY set the preference in the state file and
  report the new state — they do NOT perform a pull/upgrade.
- A manual `chosko-llm upgrade` (no auto flag) when the preference is NOT
  enabled prints a tip that the user can opt in with
  `chosko-llm upgrade --enable-auto`.
- The `CHOSKO_LLM_NO_AUTO_UPGRADE` environment variable, when set, force-
  skips the automatic run (escape hatch for CI / non-interactive use).
- `docs/cli-help.txt` (the `upgrade` entry), the README "Keeping up to
  date" section and Configuration env-var table, and
  `.claude/context/cli-entry.md` describe the new behavior. A new
  `tests/smoke/auto-upgrade.md` checklist covers: install opt-in,
  once-per-day firing, missing-file default, `--enable-auto`/`--disable-auto`
  toggling (no upgrade), the disable suggestion when it fires, the opt-in
  tip on manual upgrade, and `CHOSKO_LLM_NO_AUTO_UPGRADE`.

## Decisions
- State-file read/write helpers live in `scripts/lib.sh`; the orchestration
  (decide-and-run) lives in a dedicated `scripts/auto-upgrade.sh` invoked by
  the proxy with `|| true`. This keeps the proxy minimal (one guarded line)
  and isolates the feature; `cmd-upgrade.sh` reuses the lib helpers for the
  flags and the opt-in tip.
- "Once a day" means a calendar-day comparison (store the date, run when it
  differs from today). POSIX-portable; avoids rolling-24h epoch math.
- `--enable-auto` / `--disable-auto` are toggle-only: they set the
  preference and exit-report, and do NOT run an upgrade.
- The automatic run is not TTY-gated (it runs per the requirement even in
  non-interactive contexts), but the suggestion/tip text IS TTY-gated like
  the existing `ls` suggestions. `CHOSKO_LLM_NO_AUTO_UPGRADE` is the
  non-interactive opt-out.
- Proxy-change implication: `bin/chosko-llm` itself changes, so existing
  users only pick up the auto-upgrade hook after one `chosko-llm upgrade`
  (which refreshes the proxy from the managed clone) or after re-running
  `install.sh`. New users get it at install. This must be called out in the
  implementing commit's body.
- No frontmatter version bumps: all touched files are CLI scripts / docs,
  not versioned command/skill features. (Optionally bump the root `VERSION`
  file if the project treats it as the CLI version — confirm with current
  convention before doing so.)
