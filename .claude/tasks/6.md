# Task 6 — Task workflow — `/task-setup` offers to commit the new scaffolding at the end

## Description
Today, `commands/task-setup.md` ends after creating the backlog
scaffolding (`.claude/TASKS.md`, `.claude/tasks/`, the four files
under `.claude/external/`) and explicitly says under **DO NOT**:
"Commit. The user decides whether to commit the scaffolding." The
user must then remember to stage and commit those new files by hand.

This is the same friction `/task-add` had before Task 4 fixed it —
and the fix is the same shape: add a final, optional, explicit
commit phase. The files `/task-setup` writes are a natural,
self-contained commit (or two, since the index/body scaffolding and
the external-LLM wiring are conceptually distinct, but bundling
them is fine and matches the bias-toward-one-commit pattern from
Task 4).

The wrinkle that distinguishes `/task-setup` from `/task-add`:
`/task-setup` is idempotent. On a re-run it may write zero, one, or
several files depending on what's already in place. The commit
offer must reflect that:
- If nothing was created or modified this run, do not prompt — there
  is nothing to commit.
- If something was written, prompt and stage exactly those paths.

The command must remain safe with a dirty working tree — it must
stage ONLY the files it just wrote, never `-A`/`-u`, and never
amend or push.

### Conventions to follow
- Tool discipline already documented in `task-setup.md`: file reads via Read; file writes via Write; Bash is currently used only for `mkdir -p` and `chmod +x`. Extend the allowed-Bash scope to cover the new git phase (`git status --porcelain`, `git add -- <path>...`, `git commit`) and document it.
- Stage only the files `/task-setup` actually wrote this run. Never `git add -A`, `git add .`, or `git add -u`.
- Never use `--no-verify`, `--amend`, `--no-gpg-sign`, or skip hooks. If a pre-commit hook fails, surface the failure and let the user fix it; do not auto-retry.
- The commit prompt must be explicit and silence must mean no commit. Match `/task-add` PHASE 5's posture exactly.
- If nothing was written this run, skip the prompt entirely — no "nothing to commit" noise.
- Frontmatter `version` must bump (per CLAUDE.md hard rule + authoring-guide).

### Out of scope

- Touching files other than the ones `/task-setup` itself wrote during this run.
- Changing the existing artifact-probe / artifact-create / inference flow except to (a) track which files were written and (b) append the new commit-offer phase.
- Changing `/task-add`, `/task-clean`, `/task-implement`, `/task-list`.

- Adding any new dependencies (jq, yq, etc.) — keep within the existing POSIX-bash + Read/Write/Edit toolset.

### Behavior change
- The artifact-creation phase is unchanged in what it writes. Internally, the command now tracks the list of paths it wrote during the run (call it `WRITTEN`).
- A new **PHASE — OFFER COMMIT** runs after the report step:
  1. If `WRITTEN` is empty (fully idempotent re-run, "Backlog already initialized"), skip — no prompt, no output beyond the existing report.
  2. Otherwise, print exactly one prompt: `Scaffolding written (<N> file(s)). Commit them now? [y/N]` followed by the bullet list of paths in `WRITTEN`.
  3. On explicit yes (`y`, `yes`, `commit`, `go`):
     - Stage exactly the paths in `WRITTEN` (`git add -- <path1> <path2> ...`).
     - Create one commit with message:
       ```
       Initialize task backlog scaffolding

       <bullet list of created paths, one per line>
       ```
       Use a HEREDOC for the multi-line body. Drop the second `-m` if `WRITTEN` has only one path (the headline alone is enough).
     - Report the resulting commit hash (`git rev-parse --short HEAD`).
  4. On anything else (no, blank line, EOF, unrelated reply, silence): print `Skipped commit. Files left unstaged.` and stop. Do not stage, do not commit.
  5. On commit failure (e.g. pre-commit hook rejects): surface the exact failure output; do NOT retry, amend, or use `--no-verify`. Files remain in whatever state git left them; tell the user.
- The closing `DO NOT: Commit. The user decides whether to commit the scaffolding.` line is removed.
- The closing `DO NOT:` block gains explicit prohibitions on `git add -A`/`-u`, amend, push, hook-skipping flags, and committing without explicit approval — mirroring `/task-add`'s closing block.
- The TOOL DISCIPLINE block is updated to note that Bash is also permitted for the optional git phase, alongside the existing `mkdir -p` / `chmod +x` uses.

### Tests
- `tests/smoke/task-setup.md` — extend the manual smoke checklist with new scenarios mirroring the four canonical commit-prompt cases from `tests/smoke/task-add.md`:
  - **Commit-yes on fresh init**: run `/task-setup` on a fresh project, answer `y`, observe a single commit whose `git show --stat HEAD` lists exactly the files `/task-setup` wrote (and nothing else).
  - **Commit-no on fresh init**: run `/task-setup`, answer `n`, observe all scaffolding files are written but unstaged.
  - **Dirty-tree safety**: pre-create an unrelated modified file, run `/task-setup`, answer `y`, verify the unrelated change is NOT included in the commit.
