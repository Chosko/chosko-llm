# Task 19 — Add semantic color to chosko-llm command output

Target: claude

## Goal
Today color lives only in the `[info]/[warn]/[error]` log prefixes (stderr).
The actual command output — the `ls` table and the "Installed/Updated/Removed"
lines — is plain. Extend the existing `lib.sh` color infrastructure with a
green success channel and reusable semantic color helpers, then apply a
consistent palette across every command so status is readable at a glance,
while fully honoring `NO_COLOR` and non-TTY output.

## Acceptance criteria
- `lib.sh` gains a `log_success` helper: bold green, stderr, gated by the same
  `NO_COLOR` + TTY check as `log_info/warn/error`.
- `lib.sh` gains a stdout-aware color gate (checks `[ -t 1 ]`) so the `ls`
  table colorizes only on a TTY; `chosko-llm ls | cat` (or redirect to a file)
  yields plain text.
- `lib.sh` gains semantic color helpers (green/yellow/cyan/dim/bold/reset) that
  emit codes only when color is enabled. No raw `\033[` escapes are scattered
  across `cmd-*.sh` — they call the helpers.
- `ls`: STATUS column color-coded — up-to-date=green, updatable=yellow,
  not installed=dim, local only=cyan; header row bold; KIND and `—`
  placeholders dim.
- `add` / `rm` / `update`: success lines use green `log_success`; skip/
  "version ahead" lines use yellow `log_warn`; "already installed"/
  "already up-to-date" stay `log_info`.
- `upgrade`: "Already up to date" uses `log_success`; pull/info lines stay
  `log_info`; the `git log --oneline` output is left to git's own coloring.
- `help`: the `Usage:` heading is bolded when color is on — in both the
  `docs/cli-help.txt` path and the fallback heredoc — and plain when piped or
  under `NO_COLOR`. `docs/cli-help.txt` stays ANSI-free (cmd-help.sh applies
  the bold conditionally; codes are never embedded in the file).
- All coloring is suppressed when `NO_COLOR` is set or output is not a TTY.
- The future `show` command (task 17) adopts these same helpers when it is
  implemented — documented so the palette stays consistent.

## Decisions
- Add a dedicated green `log_success` rather than overloading blue `log_info`
  (user choice) — gives success its own unambiguous cue.
- Color the `help` Usage heading too (user choice), but do it in `cmd-help.sh`
  by conditionally bolding heading lines, NOT by embedding ANSI in
  `docs/cli-help.txt` — keeps that file plain, diffable, and `NO_COLOR`-safe.
- Gate stdout and stderr coloring separately: the `ls` table and `help` write
  to stdout while `log_*` write to stderr, so a single `[ -t 2 ]` check is
  wrong for the table.
- Centralize all escape sequences in `lib.sh` helpers; command scripts never
  inline `\033[…` — single source for the `NO_COLOR`/TTY gate.
- The `show` command (task 17, currently in progress) is out of scope to build
  here but must reuse these helpers when implemented. Not wired as a
  precondition on task 17, since this command does not renumber or edit other
  tasks; the coupling is recorded as a hint instead.
