# Task 35 — Propose splitting a /task-add description into multiple tasks when useful

Target: claude

## Goal
Today /task-add always produces exactly one task. Add a split-suggestion
step so it can propose breaking the work into multiple tasks when that would
produce better units — either because the description bundles several
independent deliverables, or because a single task is simply too big and
would be cleaner as several smaller ones. The proposal is a suggestion, not
a gate: it should not appear for work that's fine as one task. When the user
accepts, task-add writes every part in one run (sequential IDs, one body
file each) and auto-wires Preconditions between parts that depend on each
other. Declining, or passing --no-split, keeps the current single-task flow.

## Acceptance criteria
- A new step (after PHASE 1 READ, before PHASE 2 ASK) considers whether the
  work would be better as multiple tasks — because it bundles independent
  deliverables OR because a single task would be too large — and, when so,
  proposes a split. It stays quiet for work that's fine as one task.
- The proposal lists each proposed task (title + one-line scope) and lets
  the user accept, decline, or adjust the breakdown. Declining falls back to
  a single task and the normal flow continues.
- On acceptance, each part is written as its own TASKS.md summary block +
  .claude/tasks/<N>.md body file with sequential IDs; "Last task number"
  advances by the number of parts.
- Preconditions are auto-wired: a part that depends on an earlier part gets
  that earlier part's ID in its Preconditions line (else "none").
- A `--no-split` flag skips the suggestion entirely and always writes exactly
  one task. It is documented in the usage line and coexists with `--enrich`
  and `--no-commit`.
- All resulting tasks are committed in a single commit (unless `--no-commit`),
  with a message covering every task ID created.
- Single-task behavior is unchanged when no split is suggested, when the user
  declines, or when `--no-split` is passed.
- The task-add frontmatter `version:` is bumped (minor) and the root VERSION
  file is bumped.

## Decisions
- The split is a suggestion, not a gate: it surfaces when a split would give
  better units (multiple deliverables, or one oversized task) and stays quiet
  otherwise. `--no-split` forces the old single-task behavior.
- Accepted splits are written all at once with auto-wired Preconditions
  (user's choice), rather than leaving dependencies for the user to wire.
- The split step lands between READ and ASK so the suggestion uses the
  grounded file/scope picture from READ, and the ASK/DRAFT phases then
  operate per resulting task.
