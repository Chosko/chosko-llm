# Task 2 — Task workflow — `task-implement` uses the richer body as a context starting point

## Description
Task 1 makes per-task body files (`.claude/tasks/<N>.md`) much richer:
Required reading, Relevant snippets, Conventions to follow, and Out of
scope are embedded directly. The author (Claude Code via `/task-add`)
has already done the work of identifying what's relevant.

`/task-implement` should take advantage of this so Claude Code can spend
its context budget on implementation rather than rediscovering what the
body already states. The change is **a hint, not a constraint**: the
body is the natural starting point, and a competent agent reading it
often does not need to fan out into the broader context layer — but
Claude still uses its own judgment about when reading more files is
genuinely useful (unfamiliar subsystem, snippet truncated mid-function,
conflicting signals, etc.). The goal is to reduce reflexive bulk-reads,
not to discourage warranted ones.

### Behavior change

**`commands/task-implement.md`**

1. Insert a short paragraph (a single subsection, ~6–10 lines) titled
   **USING THE TASK BODY** between LOCATING THE BACKLOG and LOCATING
   THE TEST RUNNER. Wording, roughly:

   > Per-task body files authored by `/task-add` v0.2+ embed Required
   > reading, Relevant snippets, Conventions to follow, and Out of
   > scope sections. When present, treat them as a high-quality
   > starting point — they often contain enough context to implement
   > without further reading. Reach for CLAUDE.md, `.claude/context/`,
   > and other source files when they would meaningfully inform the
   > change (unfamiliar subsystem, snippet too narrow, contradictory
   > signals), and skip them when the body is already sufficient. Use
   > judgment, not a checklist. Older body files lack these sections —
   > for those, consult the context layer as before.

2. Update PRE-FLIGHT step 3 from
   "If the project has a CLAUDE.md or `.claude/` context layer, read
   what's relevant."
   to something like:
   "If the project has a CLAUDE.md, read it — it's small and global.
   Defer reading the broader `.claude/context/` layer until per-task
   Step 1 indicates a need (see USING THE TASK BODY)."

3. Update Step 1 ("Mark IN PROGRESS") — after the existing instruction
   to read the body, add a single sentence: "If the body contains
   Required reading / Relevant snippets / Conventions / Out of scope
   sections, lean on them; pull in additional files when they
   genuinely add value."

4. Do NOT modify Step 4 ("Implement"). Leave the existing
   "Read each file before editing it" rule untouched. Do not add a
   "do not read prophylactically" instruction — that's the part that
   overconstrains the agent.

5. Bump frontmatter `version` per authoring guide (minor — wording
   guidance added, no procedural changes that break older bodies).

### Doc updates
- `.claude/context/features.md` — one-sentence touch on the
  `task-implement` bullet noting it now leans on the v0.2+ body schema
  when available.
- README/authoring-guide unchanged unless they document
  `/task-implement` behavior; verify during implementation.

### Tests
- `tests/smoke/task-implement.md` — add checklist items:
  - Running `/task-implement <N>` against a task with a v0.2+ body
    yields a successful implementation + commit, and a manual scan of
    the agent's tool-call trace shows the body sections were used as
    the primary basis (i.e. the agent did not bulk-read every file in
    `.claude/context/` reflexively). Some context reads are fine —
    the test is qualitative, not "zero reads".
  - Running `/task-implement <N>` against a body lacking the new
    sections still works and behaves as before — context layer is
    consulted normally.
- Verification surface is the smoke checklist; this is a prompt-wording
  change, not a code change.

### Definition of done
- `commands/task-implement.md` contains a USING THE TASK BODY
  subsection that frames the body as a starting point, not a cage.
- PRE-FLIGHT step 3 and Step 1 reference the new guidance with
  explicit older-body fallback. Step 4 is unchanged.
- Frontmatter `version` bumped per authoring guide.
- `.claude/context/features.md` bullet for `task-implement` updated.
- `tests/smoke/task-implement.md` updated with both scenarios above.
- Manual smoke run on a v0.2+ body shows reduced — not zero —
  proactive context reads compared to the prior version.
- Full smoke-test re-run for `task-implement` passes.
