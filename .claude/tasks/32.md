# Task 32 — Document a versioning policy for the root VERSION file in CLAUDE.md

Target: claude

## Goal
The root `VERSION` file (currently `0.1.0`) is read by `install.sh` to
print the installed version, but nothing ever bumps it — it has drifted
out of sync with the repo's actual evolution. Rather than wire up tooling,
establish a lightweight convention: document in the project `CLAUDE.md`
that `VERSION` must be bumped whenever a change ships, so the policy is
enforced by the authoring workflow going forward.

## Acceptance criteria
- CLAUDE.md gains a short, unambiguous rule stating that the root `VERSION`
  file must be bumped on every change that ships.
- The rule explains the bump convention (semver: patch for fixes/docs,
  minor for new features/commands/skills, major for breaking CLI changes)
  so an author can pick the right increment without guessing.
- The rule clarifies that this repo-level `VERSION` is distinct from the
  per-feature `version:` frontmatter (which versions individual
  commands/skills) — bumping a feature's frontmatter does not replace
  bumping `VERSION`.
- README's existing versioning guidance (if any) stays consistent with the
  new rule; no contradictions remain.

## Decisions
- Documentation-only, per the user's "adding this info in CLAUDE.md should
  be enough" — no script or hook is added to auto-bump VERSION.
- VERSION is in fact already consumed by `install.sh`; the task framing is
  "not maintained," not "not used." The rule targets maintenance, not
  introducing a new consumer.

## Hints
- CLAUDE.md — add the rule under "Hard rules" (or a short "Versioning"
  subsection); this is the project-checked-in CLAUDE.md, not the global one.
- README.md — reconcile any existing version/design-rule wording.
- VERSION — reference only; do not bump it as part of this task.
