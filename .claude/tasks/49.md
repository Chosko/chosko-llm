# Task 49 — Teach /task-implement to drive Unity via MCP at manual checkpoints, gated and opt-outable

Target: claude

## Goal
On `claude+human` / `human` tasks, when the project declares a Unity MCP
plugin AND its tools are live this session, let /task-implement do the
Unity work itself instead of pausing for the user — checking the Console
after compilation, and attempting editor actions — then flip each
checkpoint into a "verify my work" step. This must cost nothing on projects
that don't have Unity MCP configured, and the user can opt out per run.

## Acceptance criteria
- New supporting file `skills/task-implement/unity-mcp-checkpoints.md`
  describing the enhanced behaviour:
  1. Console-log auto-check: a checkpoint whose only ask is "check the
     Unity Console after compilation" is performed by Claude via the
     `mcp__UnityMCP__*` tools (fully automatic when it's the sole step); on
     errors, Claude iterates to fix and recompiles before proceeding.
  2. Manual-action automation: Claude attempts the editor action itself
     (e.g. create the GameObject); on success it rewrites the checkpoint
     from an instruction into a verification ("confirm Bar now has a child
     GameObject Foo") delivered at the end per the existing explain-then-ask
     protocol; steps Claude genuinely cannot do stay manual for the user.
- The gate is cheap and lives in SKILL.md Step 1 / human-in-loop.md: only
  when CLAUDE.md contains `Unity MCP for /task-implement:` AND the
  `mcp__UnityMCP__*` tools are present this session is
  unity-mcp-checkpoints.md read. Marker-but-no-tools → today's manual
  behaviour, and a one-line pointer to /unity-mcp-setup; no expensive read.
  No marker → byte-for-byte today's behaviour.
- Per-run opt-out: when a checkpoint is MCP-eligible, the existing "proceed"
  prompt offers a plain-language choice — roughly "shall I make the changes
  in the Unity Editor myself, or pause and have you do the Unity steps
  manually?" Opting out runs the standard manual-checkpoint path.
- `skills/task-implement/human-in-loop.md` updated to branch into the
  enhanced path and to preserve the explain-then-ask / independent-verify
  protocol for the rewritten verification steps.
- SKILL.md SUPPORTING FILES table gains the new file with its exact-when
  condition; the "don't read speculatively" rule is respected.
- Parity guard (scripts/check-task-parity.sh) still passes — the new
  behaviour is Claude-only and must not desync the prompt↔cmd-task-impl.sh
  contract (the local-LLM/aider path can't drive Unity MCP; confirm it's
  unaffected or update the guard accordingly).
- features.md + README updated; task-implement frontmatter version bumped
  (minor) and root VERSION bumped (minor).

## Decisions
- Enhancement is Claude-only and does not touch the external-LLM
  (cmd-task-impl.sh / aider) implementation path.
- Gate ordering keeps the hot path free: CLAUDE.md marker check (already in
  context) → in-session tool presence → only then read the behaviour doc.
- Editor liveness isn't pre-probed: Claude attempts the MCP action and
  falls back to a manual checkpoint if the call fails (e.g. Unity closed).
