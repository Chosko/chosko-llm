# Task 1 — Task workflow — make backlog consumable by external LLMs (aider + Ollama)

## Description
Today the `task-*` commands are written assuming Claude Code is both the
*author* and the *consumer* of tasks. We want to keep Claude Code as the
author (planning + writing rich task bodies) but offload *implementation*
to a smaller local model — concretely: **qwen2.5-coder:14b via Ollama,
driven by aider**. aider gives the model file-read, repo-map, and
diff-based file-edit tools, so it can navigate a project. What it cannot
do is infer Claude-Code-specific conventions or follow a `/task-implement`
skill flow.

Two artifacts need to change so this handoff works:

1. **Per-task body files (`.claude/tasks/<N>.md`)** must be self-contained
   enough that a 14B coder model with file tools can implement from them.
   That means explicit *required reading* (file paths the model should
   `/add` to aider's context), short *quoted snippets* for non-obvious
   code, *conventions* the model must follow, *out of scope* guardrails,
   and concrete *acceptance criteria*. Today the schema has `Description`,
   `Behavior change`, `Tests`, `Definition of done` — adequate for Claude
   Code, too implicit for a weaker model.

2. **A static implement procedure** must live in each project at
   `.claude/external/implement-prompt.md`. This is the system-prompt
   analogue of `/task-implement` — it tells the external LLM how to
   consume any task body file in this project. It is project-local
   (travels via git so a teammate can clone-and-run) and produced by
   `/task-setup`. The aider invocation pattern is:
     `aider --model ollama/qwen2.5-coder:14b \`
     `  --read .claude/external/implement-prompt.md \`
     `  --read .claude/tasks/<N>.md`

### Behavior change

**`commands/task-setup.md`**
- In addition to creating `.claude/TASKS.md` and `.claude/tasks/`, also
  create `.claude/external/implement-prompt.md` with the static template
  described below.
- If `.claude/external/implement-prompt.md` already exists, leave it
  untouched (idempotent — same rule as TASKS.md). Do NOT add a `--force`
  flag in this task; that's a separate concern.
- Update the command's user-facing summary to mention the third artifact.
- Bump frontmatter `version` per the authoring guide (minor bump — new
  artifact created, backwards compatible for projects that re-run setup).

**`commands/task-add.md`**
- Extend the body-file schema with these additional sections (in this
  order, after `## Description`):

    ### Files to modify
    Replicates the `Files:` field from the task's TASKS.md summary
    block — the comma-separated list of files this task will edit.
    Lives in the body so an external LLM fed only `.claude/tasks/<N>.md`
    knows the output surface without having to parse TASKS.md. `/task-add`
    writes both copies at creation; neither changes during
    implementation, so drift risk is low. Do NOT replicate `Status:` or
    `Preconditions:` — those are backlog-flow concerns, not
    implementation context.

    ### Required reading
    Bulleted list of `path:line-range — why` entries the implementer
    should `/add` to aider before editing. Include CLAUDE.md, the
    relevant `.claude/context/*.md`, and the source files involved.

    ### Relevant snippets
    Optional. Quote 5–30 line excerpts of code that are central to the
    task and non-obvious. Each snippet prefixed with its `path:line`
    origin. Skip when the required-reading list is enough.

    ### Conventions to follow
    Bulleted list of project rules the implementer must respect (e.g.
    "scripts start with `set -euo pipefail`", "no jq/yq deps", "honor
    `CHOSKO_LLM_HOME`/`CLAUDE_HOME`"). Pull these from CLAUDE.md and
    from the context files relevant to the touched code.

    ### Out of scope
    Bulleted list of things the implementer must NOT do (e.g. "do not
    add new dependencies", "do not refactor unrelated files", "do not
    bump versions of unrelated features"). Explicit guardrails.

- Keep the existing `### Behavior change`, `### Tests`,
  `### Definition of done` sections. `Definition of done` becomes the
  aider-facing acceptance check.

- Update PHASE 1 (READ) instructions: in addition to TASKS.md and the
  files implied by the user's description, the assistant must now
  actively gather material for **Required reading**, **Relevant
  snippets**, **Conventions**, and **Out of scope**. The PHASE 3 draft
  must show all body sections including the new `### Files to modify`
  block (which mirrors the summary block's `Files:` field).

- Update PHASE 2 (ASK): if the user's description leaves any of the four
  new sections substantially empty, ask before drafting.

- Bump frontmatter `version` (minor bump — schema extended, old bodies
  still valid).

**Static template — `.claude/external/implement-prompt.md`**
The exact content is to be drafted as part of the implementation, but
must cover:
- Role: "You are an engineer implementing one task from this project's
  backlog. You are running inside aider with file-read and file-edit
  tools."
- Inputs: the task body file at `.claude/tasks/<N>.md`.
- Procedure:
  1. Read the task body in full. `/add` every file under "Required
     reading".
  2. Skim CLAUDE.md and any cited context files; honor every rule under
     "Conventions to follow".
  3. Implement the change file by file. Stop at any rule under "Out of
     scope".
  4. Add/extend tests as named under "Tests".
  5. Verify every bullet under "Definition of done" is observable.
  6. Run the project's full test suite.
- Output discipline: aider-style SEARCH/REPLACE diff blocks only; no
  speculative refactors.
- Stop conditions: ambiguity in the body, missing required-reading file,
  failing test that was not introduced by this change.

### Doc updates
- `docs/authoring-guide.md` — if it documents the per-task body schema,
  add the four new sections. (Read the file during implementation to
  confirm; it may not currently spec the body schema.)
- `.claude/context/features.md` — update the bullet for `task-add` and
  `task-setup` to reflect the new artifact list and richer schema.
- `README.md` — if it advertises the task workflow, mention the
  external-LLM handoff pattern in one line.

### Tests
- `tests/smoke/task-setup.md` — add a checklist item: after running
  `/task-setup` in a clean project, `.claude/external/implement-prompt.md`
  exists and contains the role/procedure sections. Re-running
  `/task-setup` does not overwrite an edited prompt file.
- `tests/smoke/task-add.md` — add checklist items: after running
  `/task-add <description>` against a project where `/task-setup` has
  run, the resulting `.claude/tasks/<N>.md` contains the expected
  sections (Description, Files to modify, Required reading, Conventions
  to follow, Out of scope, Tests, Definition of done — with Relevant
  snippets and Behavior change as applicable). The `Files to modify`
  section must match the `Files:` field of the same task's summary
  block in TASKS.md verbatim. Run end-to-end manually with a real
  example task.
- End-to-end aider verification (manual, documented in the task-add
  smoke test): launch aider with the qwen2.5-coder:14b model, the
  external prompt, and a real `.claude/tasks/<N>.md`; confirm aider
  can navigate to the required-reading files and produce a plausible
  diff. Pass criterion is qualitative: model identifies the right
  files and proposes edits in the right places. This is a sanity
  check, not a regression gate.

### Definition of done
- `commands/task-setup.md` creates `.claude/external/implement-prompt.md`
  in addition to existing artifacts; idempotent on re-run.
- `commands/task-add.md` produces body files with the five new sections
  (Files to modify, Required reading, Relevant snippets [optional],
  Conventions to follow, Out of scope) in addition to the existing
  sections. `Files to modify` mirrors the `Files:` field from the
  TASKS.md summary block.
- The static `implement-prompt.md` template covers role, inputs,
  procedure, output discipline, and stop conditions.
- Frontmatter `version` bumped on both modified commands per the
  authoring guide.
- `tests/smoke/task-setup.md` and `tests/smoke/task-add.md` updated.
- Affected doc files (`features.md` context entry; authoring guide
  if applicable) updated.
- Manual aider+ollama smoke test passes against a real task body.
- Full smoke-test re-run for both commands passes.
