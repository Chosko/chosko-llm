# Task 20 — Add Windows cmd/PowerShell entry point (`chosko-llm.cmd` shim)

Target: claude

## Goal
Let Windows users run `chosko-llm` from cmd.exe and PowerShell, not just
git-bash. Ship a thin `bin/chosko-llm.cmd` batch shim that locates git-bash's
`bash.exe` and forwards the command line to the existing bash proxy, so all CLI
logic stays single-sourced. `install.sh` drops the shim onto the Windows PATH
when run on Windows. The git-bash-vs-WSL filesystem split and the muted
interactive output (suggestions/colors) under those shells are documented as a
known baseline, not solved.

## Acceptance criteria
- New `bin/chosko-llm.cmd`: when `chosko-llm` is invoked from cmd.exe or
  PowerShell (resolved via `PATHEXT`), it forwards all arguments to the bash
  proxy and propagates the proxy's exit code (`exit /b %ERRORLEVEL%`).
- The shim is a thin forwarder: it calls the sibling proxy via its own
  directory (`"%~dp0chosko-llm"`), duplicating no dispatch logic, so it keeps
  working regardless of `BIN_DIR`.
- The shim auto-detects git-bash by probing common locations
  (`%ProgramFiles%\Git\bin\bash.exe`, `%ProgramFiles(x86)%\Git\bin\bash.exe`,
  `%LocalAppData%\Programs\Git\bin\bash.exe`), falling back to `where bash`. If
  no bash is found it prints a clear error to stderr and exits non-zero.
- `install.sh`, when running on Windows (`uname -s` matches
  `MINGW*`/`MSYS*`/`CYGWIN*`), copies `bin/chosko-llm.cmd` into `$BIN_DIR`
  alongside the proxy, backing up any existing `chosko-llm.cmd` the same way it
  backs up the proxy. On non-Windows, install.sh behaviour is unchanged.
- install.sh prints Windows-specific guidance: `$BIN_DIR` must be on the
  **Windows** PATH (not just the MSYS PATH) for cmd/PowerShell to resolve the
  shim, shown as a native path (via `cygpath -w` when available). It instructs;
  it does not auto-modify the user's persistent PATH.
- `.gitattributes` keeps `*.cmd` as CRLF (`text eol=crlf`) so the batch file is
  well-formed on Windows.
- README documents the new entry point and the two caveats: git-bash is the
  supported bash (WSL users run from inside WSL, since `~/.chosko-llm` resolves
  to a different filesystem), and interactive suggestions/colors may be muted
  when launched from cmd/PowerShell without a pty.
- Running `chosko-llm ls` and other subcommands from cmd.exe and PowerShell
  yields the same output/behaviour as from git-bash.

## Decisions
- One `.cmd` file, no separate `.ps1`: both cmd.exe and PowerShell resolve
  `chosko-llm` → `chosko-llm.cmd` via `PATHEXT`, so a single shim covers both.
- The shim forwards to the existing bash proxy instead of reimplementing the
  subcommand dispatch — single source of truth; future CLI logic rides along.
- git-bash is the supported target. The `where bash` fallback can resolve WSL's
  `C:\Windows\System32\bash.exe`, which has a different `$HOME`/managed clone —
  so detection prefers explicit git-bash paths first and the WSL divergence is
  documented rather than solved.
- install.sh instructs about the Windows PATH rather than running `setx` —
  mutating the user's persistent PATH is invasive and matches the existing
  MSYS-PATH guidance behaviour.
- Ships by **re-running install.sh**, not `chosko-llm upgrade`: it changes
  `bin/` content and adds a new file that must land in `BIN_DIR`. Note this in
  the commit.
- Muted TTY output (task 18 suggestions, task 19 colors) under cmd/PowerShell
  without a pty is accepted as a documented baseline, not worked around.
